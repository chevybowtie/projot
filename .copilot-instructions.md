# Copilot Instructions for projot

> **Note:** This file is superseded by `CLAUDE.md` (technical guidance for Claude Code) and `INSTRUCTIONS.md`. Content below is retained for reference but should not be updated. All AI assistant guidance should be added to CLAUDE.md instead.

When assisting with projot development, follow these directives:

## Requirements (Non-Negotiable)

1. **NO external dependencies** — reject any suggestion to add libraries. Use only C++ standard library.
2. **C++17 only** — no C++20 or later syntax.
3. **All code changes require unit tests** — ask for tests if none are provided.
4. **Cross-platform** — code must compile on both Linux and Windows without modification (use `<filesystem>`, not platform-specific APIs).
5. **CMake-only builds** — any changes to the build system must use CMake, not shell scripts or Make.

## When Reviewing or Writing Code

- ✅ **Prefer standard library**: `std::vector`, `std::string`, `std::map`, `std::optional`, `std::filesystem`
- ✅ **Test-driven**: Write tests before or alongside code
- ✅ **RAII pattern**: Resources acquired in constructor, released in destructor
- ✅ **const correctness**: Mark parameters and methods const where appropriate
- ✅ **Update docs**: Modify README.md if CLI, config format, or commands change

## Forbidden Patterns

- ❌ `#include <boost/...>`, `#include <fmt/...>`, `#include <nlohmann/json.hpp>`, `#include <curl/curl.h>`, etc.
- ❌ `std::thread` or platform-specific threading (unless absolutely necessary)
- ❌ `system()` calls or shell command execution
- ❌ Hardcoded absolute paths — use `std::filesystem::absolute()` and `std::filesystem::home_directory()`
- ❌ Windows-only or Linux-only code without the other platform's equivalent

## Testing Expectations

- All new functions/classes must have unit tests in `tests/`
- Bug fixes must include a regression test
- Run: `cmake --build build --target test` before recommending completion
- Tests must pass on both platforms (or document platform-specific behavior)

## Command & Config Stability

- Command names and arguments are **stable** — breaking changes need a major version bump
- Config file format changes require backward compatibility or versioning
- Always update README.md if user-facing behavior changes
