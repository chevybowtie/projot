# projot Development Standards

## Core Principles

This project has strict requirements to maintain quality, portability, and simplicity:

### 1. No External Dependencies
- **C++ standard library only** — use `<vector>`, `<string>`, `<map>`, `<iostream>`, `<fstream>`, `<filesystem>`, etc.
- **No third-party libraries** (no Boost, Fmt, nlohmann/json, etc.)
- Rationale: keeps the codebase portable and easy to build anywhere

### 2. Full Test Coverage
- **All new features must include unit tests**
- **All bug fixes must include tests** that would have caught the bug
- Test files go in `tests/` directory
- Use the existing test infrastructure (CMake `enable_testing()`)
- Run tests before committing: `cmake --build build --target test`

### 3. Cross-Platform (Linux + Windows)
- **No platform-specific code** unless absolutely necessary (with `#ifdef _WIN32` / `#ifdef __linux__`)
- **Use `<filesystem>`** for all path operations (not `boost::filesystem` or platform APIs)
- **Avoid POSIX-only functions** (`fork`, `unlink`, etc.)
- Test both platforms before merge

### 4. C++17 Standard
- Minimum C++17 required (`CMakeLists.txt` enforces `CMAKE_CXX_STANDARD 17`)
- Use modern C++ features: structured bindings, `std::optional`, `std::variant`, range-based for loops
- No C++20 features

### 5. CMake Build System
- Build with: `cmake -B build && cmake --build build`
- All targets must be specified in `CMakeLists.txt`
- No Makefiles, shell scripts, or custom build steps outside CMake

### 6. CLI Stability
- **Command names and arguments are stable** — breaking changes require a major version bump
- New flags can be added (`--new-flag`), but don't remove or rename existing ones
- Maintain backward compatibility in config file format
- Document all command changes in README.md

### 7. File Format Stability
- The `.projot/config` and notes file (`.projot/{RANP}.md`) formats are stable
- Migrations required if format changes
- Document any format version changes

## Code Style

- Follow C++ standard conventions (naming: `snake_case` for functions, `CamelCase` for classes)
- Use RAII (Resource Acquisition Is Initialization)
- Prefer `const` correctness
- Use `auto` where type is obvious; explicit types for clarity otherwise
- Single responsibility principle: functions do one thing well

## Workflow

1. **Branch** from `main` for new features/fixes
2. **Write tests first** (or at least alongside code)
3. **Build and test**: `cmake --build build --target test`
4. **No external dependencies** — code review will reject these
5. **Update README.md** if commands, config, or behavior changes
6. **Create PR** with clear description of what changed and why

## Red Flags (Will Be Rejected)

- ❌ External dependencies (`#include <nlohmann/json.hpp>`, `#include <fmt/format.h>`, etc.)
- ❌ Missing unit tests for new code
- ❌ Windows-specific code without Linux equivalent
- ❌ C++20 or later features
- ❌ Breaking changes to CLI without major version bump
- ❌ Hardcoded paths (use `std::filesystem`)
- ❌ Platform detection without handling both Linux and Windows
